Database API Concept Notes

Learn how API design meets persistence, transactions, query safety, and performance-aware data access. ← Back to Lab · Practice Page

1. What a Database API is

A Database API is the part of a backend that exposes data persistence safely and intentionally. It might use raw SQL, an ORM, a query builder, or a repository pattern — but the goal is the same: turn business actions into correct database reads and writes.

The trick is not merely “connect to database.” The real skill is deciding how much query power to expose, how to validate inputs, and how to preserve consistency under load.

2. Core building blocks

Concept Why it matters
Models/entities Represent domain records like users, products, and orders.
Relationships Handle one-to-many, many-to-many, and ownership rules.
Migrations Version schema changes over time.
Transactions Protect multi-step writes from partial failure.
Indexes Improve query performance for frequent lookups.
Pagination Protect APIs from huge scans and payloads.

3. Query safety

Never build SQL with unchecked string concatenation. Use parameterized queries or ORM-safe bindings. 

// good
SELECT * FROM products WHERE category = ? AND price >= ?

// dangerous
"SELECT * FROM products WHERE category = '" + category + "'"

APIs that expose filtering, sorting, and search must carefully validate allowed fields so “flexible” does not become “oops, free SQL adventure.”

4. Transactions and consistency

Whenever several writes must succeed or fail together, use a transaction. 

BEGIN;
  INSERT INTO orders (...);
  INSERT INTO order_items (...);
  UPDATE inventory SET stock = stock - 1 WHERE product_id = ?;
COMMIT;

-- if anything fails
ROLLBACK;
If your API changes money, inventory, booking state, or security records, transaction design is not optional.

5. Performance concepts

  • Use indexes for common filters and joins.
  • Paginate large collections.
  • Select only needed columns.
  • Watch for N+1 query problems in ORMs.
  • Measure slow queries before guessing at fixes.

6. ORM vs raw SQL

Approach Strength Weakness
ORM Fast development, model mapping, migrations Can hide bad queries
Query builder More explicit SQL control More verbose
Raw SQL Maximum precision and performance More manual plumbing
